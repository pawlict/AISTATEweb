AISTATEweb Dokumentacja Techniczna — wersja 3.6 beta

AISTATEweb to zintegrowana platforma analityczna oparta na sztucznej inteligencji, zaprojektowana do transkrypcji audio, diaryzacji mówców, tłumaczenia dokumentów, analizy bilingów GSM, wykrywania prania pieniędzy (AML) oraz interakcji z lokalnymi modelami LLM. System działa w pełni lokalnie — bez przesyłania danych do chmury.

Kluczowe możliwości

  • Transkrypcja audio — Whisper (OpenAI), NeMo (NVIDIA), obsługa 99+ języków
  • Diaryzacja głosów — pyannote.audio, automatyczne rozpoznawanie mówców
  • Detekcja dźwięków — YAMNet, PANNs, BEATs (rozpoznawanie odgłosów otoczenia)
  • Tłumaczenie — NLLB-200 (Meta, 200+ języków) + LLM hybrydowe
  • Chat LLM — Ollama (lokalne modele AI), strumieniowanie SSE
  • Analiza bilingów GSM — Plus, Play, T-Mobile, Orange, Orange Retencja
  • AML — Analiza wyciągów bankowych, wykrywanie prania pieniędzy
  • Analiza finansowa — 7 parserów bankowych, kategoryzacja, scoring
  • Synteza mowy (TTS) — Piper, MMS, Kokoro
  • Multi-user — System ról RBAC, workspace'y, audyt

Przepływ danych

  1. Interfejs webowy — przeglądarka wysyła żądania HTTP do FastAPI
  2. Router — serwer kieruje do odpowiedniego modułu backend
  3. TaskManager — ciężkie zadania ML (transkrypcja, tłumaczenie, TTS) uruchamiane jako subprocesy Python
  4. GPUResourceManager — kolejkuje zadania GPU z priorytetami, zapobiega wyczerpaniu VRAM
  5. Workers — raporują postęp przez PROGRESS: N na stderr, wyniki jako JSON na stdout
  6. Baza danych — SQLite w trybie WAL (współbieżne odczyty), 27 tabel, 47 indeksów

7. Moduł: Transkrypcja audio

Opis

Automatyczna konwersja nagrań audio na tekst z wykorzystaniem modeli ASR (Automatic Speech Recognition).

Funkcjonalności

  • Silniki: Whisper (OpenAI, 5 rozmiarów: tiny–large), NeMo (NVIDIA Conformer CTC)
  • Obsługa 99+ języków z automatyczną detekcją
  • Segmentacja z timestampami (YYYY-MM-DD HH:MM:SS)
  • Edycja segmentów inline (split, merge, text edit)
  • Wizualizacja waveform (Web Audio API)
  • Odtwarzacz audio z synchronizacją segmentów

12b. Adaptywny system parserów GSM

Problem

Operatorzy GSM (Plus, Play, T-Mobile, Orange) okresowo zmieniają formaty swoich bilingów — przenoszą kolumny, zmieniają ich nazwy, dodają nowe pola. Parsery oparte na dokładnym dopasowaniu nazw kolumn (Plus, Play) przestają działać poprawnie po takiej zmianie — operator jest rozpoznawany (detekcja regex działa), ale mapowanie kolumn nie trafia i użytkownik dostaje 0 rekordów.

12b.1 Odporność parserów na zmiany formatów

OperatorMechanizm mapowaniaOdpornośćAdaptive fallback
Plus (Polkomtel)_POL_COLUMNS.get(header_lower) — dokładny dict lookupKruchyTAK
Play (P4)h == csv_lower — dokładne porównanie stringBardzo kruchyTAK
T-Mobilebuild_column_map() z regex patternsOdpornyNie potrzebny
Orangebuild_column_map() z regex patternsOdpornyNie potrzebny
Orange Retencjabuild_column_map() z regex patternsOdpornyNie potrzebny

12b.2 Architektura 5-warstwowa

Warstwa 1: Schema Registry        — JSON z "known good" schematami kolumn
Warstwa 2: Schema Validator        — porównanie nagłówków vs schemat (exact → regex → fuzzy)
Warstwa 3: Adaptive Column Mapper  — fuzzy fallback gdy exact match fails
Warstwa 4: Drift Reporter          — zapis/raportowanie wykrytych zmian formatu
Warstwa 5: Parser Updater          — modyfikacja kodu parsera z backup + aprobata użytkownika

Każda warstwa jest niezależna. Istniejące parsery działają bez zmian — adaptywna warstwa opakowuje je i interweniuje wyłącznie gdy standardowa ścieżka mapowania nie trafia w krytyczne kolumny.

12b.3 Warstwa 1: Schema Registry

backend/gsm/parsers/schema_registry.py

  • ColumnSchema — dataclass: logical_name, expected_headers (list), regex_patterns (list), required (bool), data_type
  • ParserSchema — dataclass: parser_id, parser_version, format_variant, columns (list), header_detection
  • SchemaRegistry — zapis/odczyt schematów JSON w katalogu schemas/
  • Auto-bootstrapping — przy pierwszym użyciu generuje schematy automatycznie z istniejących _COLUMNS dict parserów (zero pracy ręcznej)
  • Obsługuje zarówno exact-match dict (Plus/Play) jak i regex-pattern dict (T-Mobile/Orange)

Generowane schematy per parser/wariant: plus_POL, plus_TD, play, tmobile, orange, orange_retencja, generic

12b.4 Warstwa 2: Schema Validator

backend/gsm/parsers/schema_validator.py

Trójfazowa walidacja nagłówków pliku bilingowego:

  1. Exact match — porównanie z expected_headers (obecne zachowanie parserów)
  2. Regex match — użycie regex_patterns (jak build_column_map() w T-Mobile/Orange)
  3. Fuzzy match — 3 strategie dopasowania rozmytego:
StrategiaMetodaPrzykład
Normalizacja UnicodeUsunięcie polskich diakrytyków (ł→l, ą→a, ś→s...), zamiana _ na spacje, zwinięcie whitespaceDATA_I_GODZ_POLACZ ↔ Data i godz. polacz. → 1.0
SequenceMatcherdifflib.SequenceMatcher.ratio() ≥ 0.55czas trwania ↔ czas_trwania → 0.92
SemantycznySłownik równoważników polskiej terminologii bilingowej (11 grup semantycznych)data rozpoczecia ↔ datetime → match via grupa "datetime"

Grupy semantyczne polskiej terminologii bilingowej

Wynik walidacji (SchemaValidationResult):

  • match_type: "exact" | "partial" | "drift" | "failed"
  • matched_columns: Dict — zmapowane kolumny z indeksami
  • missing_columns: List — brakujące wymagane kolumny
  • fuzzy_matches: Dict — dopasowania rozmyte z confidence i metodą
  • confidence: float — ogólna pewność dopasowania (0.0–1.0)

12b.5 Warstwa 3: Adaptive Column Mapper

backend/gsm/parsers/adaptive_mapper.py

Punkt integracji z istniejącymi parserami. Aktywuje się wyłącznie gdy standardowe mapowanie nie trafia w krytyczne kolumny. Hooki w parserach:

# Plus (plus.py) — po standardowym _POL_COLUMNS.get() loop:
if "start" not in col_map or "type" not in col_map:
    mapper = AdaptiveColumnMapper()
    col_map, validation = mapper.build_adaptive_col_map(
        "plus", "POL", header_lower, col_map
    )

# Play (play.py) — po standardowym h == csv_lower loop:
if {"datetime","ui_msisdn","uw_msisdn","service_type"} - set(col_map.keys()):
    mapper = AdaptiveColumnMapper()
    col_map, validation = mapper.build_adaptive_col_map(
        "play", "", header_lower, col_map
    )

Bezpieczeństwo: Cały adaptive hook opakowany w try/except Exception: pass — warstwa adaptywna nigdy nie blokuje parsowania. Jest czysto addytywna.

12b.8 Flow użytkownika — przykład

  1. Plus zmienia nazwy kolumn w nowym bilingu (np. "DATA_I_GODZ_POLACZ" → "Data i godzina połączenia")
  2. Użytkownik uploaduje plik → /api/gsm/parse
  3. Detekcja: Plus rozpoznany (regex patterns nadal trafiają)
  4. Mapping: _POL_COLUMNS.get() nie trafia na 3 kolumny → col_map niekompletny
  5. Adaptive fallback: fuzzy matcher znajduje dopasowania:
    "typ uslugi" → type (confidence 0.88, normalizacja)
    "data rozpoczecia" → start (confidence 0.72, semantyczny)
  6. Parsowanie: kontynuuje z augmented col_map → rekordy sparsowane
  7. Drift report: zapisany z fuzzy_matches, status="pending"
  8. Odpowiedź API: zawiera ostrzeżenia o wykrytych zmianach formatu
  9. Użytkownik przechodzi do panelu dryfu → widzi raport
  10. Zatwierdza mapowania → POST .../approve z apply_to_parser: true
  11. System robi backup plus.py, dodaje wpisy do _POL_COLUMNS, bumpuje wersję → 1.4
  12. Następny upload tego samego formatu → działa natychmiast bez adaptive (exact match)

15. Uwierzytelnianie

15.1 Tryby wdrożenia

TrybOpis
singleBez logowania — pełny dostęp dla jednego użytkownika
multiPełne uwierzytelnianie, sesje, RBAC, audyt

15.2 Hashowanie haseł

Algorytm: PBKDF2-HMAC-SHA256

Iteracje: 260 000

Sól: 16 bajtów losowych

Format: pbkdf2:<iterations>:<hex_salt>:<hex_hash>

Porównanie: hmac.compare_digest() (timing-safe)

15.3 Sesje

  • Tokeny: secrets.token_urlsafe(32) — 256 bitów entropii
  • Przechowywanie: SQLite (auth_sessions)
  • Cookie: aistate_session, httponly, samesite=lax, secure (dynamicznie)
  • Timeout: 8h (konfigurowalny)
  • Walidacja: server-side na każdym request

15.4 Ochrona brute-force

  • Rate limiting IP: 5 prób / 60 sekund (in-memory)
  • Account lockout: Po 5 nieudanych próbach → blokada na 15 minut (konfigurowane)
  • Alert admin: Przy 10+ nieudanych próbach → wiadomość Call Center do adminów

15.5 Odzyskiwanie konta

  • Fraza odzyskiwania: 12 słów BIP-39 (~132 bity entropii)
  • Generowana przy rejestracji, wyświetlana raz, hashowana PBKDF2
  • Pozwala na reset hasła bez logowania

15.6 Polityki haseł

PolitykaWymagania
noneTylko sprawdzenie blacklisty
basicMin. 8 znaków
mediumMin. 8 znaków + małe + wielkie + cyfra
strongMin. 12 znaków + małe + wielkie + cyfra + znak specjalny

Konta administratorów zawsze wymuszają politykę strong.

17. Zarządzanie użytkownikami

17.1 Cykl życia użytkownika

  1. Rejestracja — użytkownik tworzy konto (pending=True)
  2. Zatwierdzenie — admin z rolą "Strażnik Dostępu" zatwierdza
  3. Aktywny — pełny dostęp zgodny z rolą
  4. Blokada — ban (tymczasowy lub stały) z powodem
  5. Usunięcie — kaskadowe usunięcie sesji i danych

17.2 Workspace'y

  • Workspace — kontener projektów z właścicielem, członkami, rolami
  • Podprojekty — potomne elementy (transkrypcja, diaryzacja, analiza...)
  • Role workspace — owner, manager, editor, commenter, viewer
  • Zaproszenia — flow: pending → accepted/rejected/cancelled
  • Log aktywności — audit trail akcji w workspace
